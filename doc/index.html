Transformenator (Download)

Transformenation is something that should be possible to do with some rudimentary shell scripting. You should be able to run a binary file through sed or awk and have byte sequences change to different byte sequences.

But you can't.

You could probably run a file through a hex dumper, change hex values, then un-transform it back to binary. But that's kind of a pain too. The problem is that sed and awk work on lines, defined as things that are delineated by what they consider line ending characters like 0x0d or 0x0a. But what if your data stream contains 0x0d and 0x0a bytes - but you don't want them to count as line endings? What if you need to remove nulls, hex zeroes, or whatever you want to call them from a binary file or data stream?

Your're stuck.

Maybe that's why you're here. Transformenator can help.

Useage

Invoke the transformenator Java jar file from any command line, ant script, or what have you. A set of rules (comprising a transform) are applied to the input file and written to the output file. Several sample transforms are included in the distribution jar file, and you can write your own as simply as creating a text file.

For example:

java -jar transformenator.jar transform infile.bin outfile.bin

Where:
transform is a file containing the set of transformations you want to make
infile.bin is the original file
outfile.bin is the resulting file after running the transforms

Example

Say you have the file below named infile.bin, represented in a typical hex editor: (Offset) (Hex data) (ASCII representation) 0000000: 6162 6364 0000 6566 6768 696a 1a67 6574 abcd..efghij.get 0000010: 2072 6964 206f 6620 6d65 210a rid of me!. Say we want to change the hex zeroes in the middle to spaces, and eliminate anything after the EOF (0x1a). Create a transform file named transform to do that: ; Swap spaces for nulls 00 = 20 ; EOF really means EOF 1a = "{@@<FiLe_EoF>@@}" So, we run infile.bin through the transform file and send the output to a file named outfile.bin: %java -jar transformenator.jar transform infile.bin outfile.bin The resulting outfile.bin looks like this in a hex editor: (Offset) (Hex data) (ASCII representation) 0000000: 6162 6364 2020 6566 6768 696a abcd efghij

Transform Specifications

The Transformenator project comes with several example transforms used to convert various early binary word processor formats to RTF, HTML, or plain ASCII text. A sample transform file might simply look like this: ; Change 0x00 to 0x20 00 = 20 Semicolons denote comment lines. The left side of the equals sign is one of these things:
  • A byte pattern, given in even numbers of hex digits, with no other decoration
    0d0a32ea4f
  • A byte pattern combined with "don't care" values consisting of even numbers of non-hex digits
    0d0a....4f
  • head specifies a beginning-of-file preamble to add to the beginning of the resultant file
    ; Header for HTML format head = "<html>"
  • tail specifies a end-of-file suffix to add to the end of the resultant file
    ; Trailer for HTML format tail = "</html>"
  • trim-leading specifies a number of bytes to trim off the beginning of the file (in hex bytes) before any binary transforms are applied
    ; Trim the leading 0x400 bytes - this is a Pascal format file trim_leading = 0400
  • regex specifies a regular expression to run on the output after any and all binary transforms are applied; multiple regex expressions are allowed, one per line
    ; Remove left/right margin specs: zlmxx, :rmxx regex = @[z|:][l|r]m[0-9]*@@ ; Remove next file specification regex = @znx:.*@@
The right side can be blank, another byte pattern given in undecorated hex, or ASCII text if surrounded in double quotes. Examples: ; 0x86 appeared at the beginning of files sometimes, after the preamble. Remove it. 86 = ; Start of heading 83 = "\par\par\pard\s2\b\fs36 " ; Italics on b12d31 = "\i " ; Italics off b12d30 = "\i0 " There is a special case for representing the end of a file. When specifying the byte pattern that signifies the end of file marker, the right side must be the production in the example below: ; EOF is ctrl-Z 1a = "{@@<FiLe_EoF>@@}" There is a special case for specifying returns and newlines. If you need to translate a byte pattern into something that includes them, you need to escape the backslash. For example: ; Justify on 7a0a1531 = "\\r\\n</pre>" Regular expressions are surrounded by delimiters - the first character in the string serves as that delimiter. So the production regex=$abc$123$ will use $ as the delimiter, and replace all occurrences of abc with 123.
